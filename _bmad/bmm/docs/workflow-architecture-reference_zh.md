# 决策架构工作流 - 技术参考

**模块：** BMM (BMAD 方法模块)
**类型：** 解决方案设计工作流

---

## 概述

决策架构工作流是对 BMAD 方法中架构决策制定方式的彻底重新构想。与模板驱动的文档不同，此工作流促进智能对话，产生**以决策为中心的架构文档**，经优化以防止实施期间的 AI 代理冲突。

---

## 核心理念

**问题**：当多个 AI 代理实施系统的不同部分时，他们会做出冲突的技术决策，导致不兼容的实施。

**解决方案**：一种"一致性契约"，预先记录所有关键技术决策，确保每个代理遵循相同的模式并使用相同的技术。

---

## 主要特点

### 1. 启动模板智能 ⭐ 新功能

- 发现相关的启动模板（create-next-app、create-t3-app 等）
- 选择模板时考虑 UX 需求（动画、可访问性等）
- 搜索当前 CLI 选项和默认值
- 记录由启动模板做出的决策
- 围绕启动基础进行剩余的架构决策
- 第一个实施故事变为"使用启动命令初始化"

### 2. 自适应促进

- 根据用户技能水平调整对话风格（初学者/中级/专家）
- 专家获得快速、技术性的讨论
- 初学者接受教育并免受复杂性影响
- 每个人都产生相同的高质量输出

### 3. 动态版本验证

- 永不信任硬编码的版本号
- 使用 WebSearch 查找当前的稳定版本
- 在对话期间验证版本
- 仅记录经过验证的当前版本

### 4. 智能发现

- 没有僵化的项目类型模板
- 分析 PRD 以确定对 THIS 项目重要的决策
- 使用决策和模式的知识库
- 扩展到无限的项目类型

### 5. 协作决策

- 促进每个关键决策的讨论
- 展示具有权衡的选项
- 集成高级启发以获得创新方法
- 确保决策是连贯且兼容的

### 6. 一致的输出

- 对话期间的结构化决策收集
- 从收集的决策生成严格文档
- 根据硬性要求进行验证
- 为 AI 代理消费进行优化

---

## 工作流结构

```
步骤 0：验证工作流并提取项目配置
步骤 0.5：验证工作流排序
步骤 1：加载 PRD（带 FR/NFR）并理解项目上下文
步骤 2：发现和评估启动模板 ⭐ 新功能
步骤 3：调整促进风格并识别剩余决策
步骤 4：促进协作决策（带版本验证）
步骤 5：处理横切关注点
步骤 6：定义项目结构和边界
步骤 7：设计新颖架构模式（需要时）⭐ 新功能
步骤 8：定义实施模式以防止代理冲突
步骤 9：验证架构连贯性
步骤 10：生成决策架构文档（带初始化命令）
步骤 11：验证文档完整性
步骤 12：最终审查和更新工作流状态
```

---

## 此工作流中的文件

- **workflow.yaml** - 配置和元数据
- **instructions.md** - 自适应促进流程
- **decision-catalog.yaml** - 所有架构决策的知识库
- **architecture-patterns.yaml** - 从需求识别的常见模式
- **pattern-categories.csv** - 模式原则，教导 LLM 需要定义什么
- **checklist.md** - 输出文档的验证要求
- **architecture-template.md** - 最终文档的严格格式

---

## 与旧架构工作流的不同之处

| 方面               | 旧工作流                                 | 新工作流                                    |
| -------------------- | -------------------------------------------- | ----------------------------------------------- |
| **方法**         | 模板驱动                              | 对话驱动                             |
| **项目类型**    | 11 种僵化类型，22+ 个文件                | 无限灵活性，智能发现 |
| **用户交互** | 输出部分带"继续？"             | 协作决策促进             |
| **技能适应** | 一刀切                            | 适应初学者/中级/专家          |
| **决策制定**  | 流程后期（步骤 5）                     | 前期和核心关注                       |
| **输出**           | 多个文档，包括虚假技术规格 | 单个以决策为中心的架构            |
| **时间**             | 令人困惑且缓慢                           | 30-90 分钟，取决于技能水平          |
| **启发**      | 从未使用                           | 在决策点集成                   |

---

## 预期输入

- **PRD**（产品需求文档）包含：
  - 功能需求
  - 非功能需求
  - 性能和合规需求

- **UX 规格**（可选但有价值）包含：
  - 界面设计和交互模式
  - 可访问性需求（WCAG 级别）
  - 动画和过渡需求
  - 平台特定的 UI 需求
  - 交互的性能期望

---

## 输出文档

单个 `architecture.md` 文件包含：

- 执行摘要（2-3 句话）
- 项目初始化命令（如果使用启动模板）
- 决策摘要表，带验证版本和史诗映射
- 完整项目结构
- 集成规格
- AI 代理的一致性规则

## 新颖模式设计如何工作

步骤 7 处理需要发明独特或复杂的模式：

### 1. 检测

工作流分析 PRD 中没有标准解决方案的概念：

- 新颖的交互模式（例如，"滑动匹配"当 Tinder 不存在时）
- 复杂的多史诗工作流（例如，"病毒式邀请系统"）
- 独特的数据关系（例如，Facebook 之前的"社交图谱"）
- 新范式（例如，Snapchat 之前的"临时消息"）

### 2. 设计协作

不仅仅是选择技术，工作流帮助设计解决方案：

- 识别要解决的核心问题
- 与用户探索不同的方法
- 记录组件如何交互
- 为复杂流程创建序列图
- 使用启发寻找创新解决方案

### 3. 文档

新颖模式成为架构的一部分，包含：

- 模式名称和目的
- 组件交互
- 数据流图
- 哪些史诗/故事受到影响
- 代理的实施指导

### 4. 示例

```
PRD："用户可以创建具有重叠成员资格的朋友'圈子'"
↓
工作流检测：这是一种新颖的社交结构模式
↓
与用户设计：圈子成员模型、权限级联、UI 模式
↓
文档："圈子模式"及组件设计和数据流
↓
所有代理理解如何一致地实施圈子相关功能
```

## 实施模式如何工作

步骤 8 通过定义一致性模式来防止代理冲突：

### 1. 核心原则

> "任何时候多个代理可能以不同方式做出相同的决策，那就是要捕获的模式"

LLM 询问："代理可能遇到什么他们必须猜测的情况？"

### 2. 模式类别（原则，而非处方）

- **命名**：如何命名事物（API、数据库字段、文件）
- **结构**：如何组织事物（文件夹、模块、层）
- **格式**：如何格式化数据（JSON 结构、响应）
- **通信**：组件如何交谈（事件、消息、协议）
- **生命周期**：状态如何变化（工作流、转换）
- **位置**：事物放在哪里（URL、路径、存储）
- **一致性**：横切关注点（日期、错误、日志）

### 3. LLM 智能

- 使用原则识别 7 个类别之外的模式
- 弄清楚所选技术的特定模式重要
- 仅询问可能导致冲突的模式
- 跳过技术选择确定的明显模式

### 4. 示例

```
选择技术：REST API + PostgreSQL + React
↓
LLM 识别需求：
- REST：URL 结构、响应格式、状态代码
- PostgreSQL：表命名、列命名、FK 模式
- React：组件结构、状态管理、测试位置
↓
与用户促进每个需求
↓
在架构中记录为实施模式
```

## 启动模板如何工作

当工作流检测到具有启动模板的项目类型时：

1. **发现**：根据 PRD 搜索相关的启动模板
2. **调查**：查找当前 CLI 选项和默认值
3. **展示**：向用户展示启动提供的内容
4. **集成**：将启动决策记录为"由启动提供"
5. **继续**：仅询问启动未做出的决策
6. **文档**：在架构中包含确切的初始化命令

### 示例流程

```
PRD 说："带有身份验证的 Next.js Web 应用程序"
↓
工作流找到：create-next-app 和 create-t3-app
↓
用户选择：create-t3-app（包括身份验证设置）
↓
启动提供：Next.js、TypeScript、tRPC、Prisma、NextAuth、Tailwind
↓
工作流仅询问：数据库选择、部署目标、其他服务
↓
第一个故事变为："npx create t3-app@latest my-app --trpc --nextauth --prisma"
```

## 用法

```bash
# 在你的 BMAD 项目中
workflow architecture
```

AI 代理将：

1. 加载你的 PRD（带 FR/NFR）
2. 识别所需的关键决策
3. 促进每个决策的讨论
4. 生成综合架构文档
5. 验证完整性

---

## 设计原则

1. **促进而非处方** - 引导用户做出良好决策，而不是强加模板
2. **智能而非模板** - 使用 AI 理解而非僵化结构
3. **决策而非细节** - 专注于防止代理冲突，而非实施细节
4. **适应而非统一** - 在确保质量输出的同时满足用户需求
5. **协作而非输出** - 对话与文档同样重要

---

## 面向开发者

此工作流假设：

- 单个开发人员 + AI 代理（而非团队）
- 速度很重要（决策在数分钟而非数天内完成）
- AI 代理需要清晰约束以防止冲突
- 架构文档是给代理的，而不是给人类的

---

## 从旧架构工作流迁移

使用旧 `architecture` 工作流的项目应该：

1. 完成任何进行中的架构工作
2. 对新项目使用 `architecture`
3. 旧工作流仍然可用但已弃用
