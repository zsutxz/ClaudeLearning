# 游戏简报工作流程

## 概述

游戏简报工作流程是BMad方法中游戏项目的起点。这是一个轻量级的、交互式的头脑风暴和规划会话，在深入详细的游戏设计文档（GDD）之前捕捉您的游戏愿景。

## 目的

**游戏简报回答：**

- 您在制作什么游戏？
- 它为谁而制作？
- 是什么让它独一无二？
- 它是否可行？

**这不是：**

- 完整的游戏设计文档
- 技术规范
- 制作计划
- 详细的内容大纲

## 何时使用此工作流程

在以下情况下使用游戏简报工作流程：

- 从头开始新的游戏项目
- 在承诺之前探索游戏想法
- 向团队/利益相关者推介概念
- 验证市场匹配度和可行性
- 为GDD工作流程准备输入

在以下情况下跳过：

- 您已经有完整的GDD
- 继续现有项目
- 无需规划需求的原型制作

## 工作流程结构

### 交互模式（推荐）

协作完成每个部分：

1. 游戏愿景（概念、演讲、愿景声明）
2. 目标市场（受众、竞争、定位）
3. 游戏基础（支柱、机制、体验目标）
4. 范围和约束（平台、时间线、预算、团队）
5. 参考框架（灵感、竞争对手、差异化点）
6. 内容框架（世界、叙事、内容量）
7. 艺术与音频方向（视觉和音频风格）
8. 风险评估（风险、挑战、缓解）
9. 成功标准（MVP、指标、发布目标）
10. 下一步行动（立即行动、研究、问题）

### YOLO模式

AI生成完整草稿，然后您迭代完善各个部分。

## 关键特性

### 可选输入

工作流程可以包含：

- 市场研究
- 头脑风暴结果
- 竞争分析
- 设计笔记
- 参考游戏列表

### 现实的范围规划

工作流程积极帮助您：

- 识别范围与资源的不匹配
- 评估技术可行性
- 识别市场风险
- 规划缓解策略

### 清晰的交接

输出设计为直接输入：

- GDD工作流程（2-规划阶段）
- 原型制作决策
- 团队讨论
- 利益相关者演示

## 输出

**game-brief-{game_name}-{date}.md** 包含：

- 执行摘要
- 完整游戏愿景
- 目标市场分析
- 核心游戏定义
- 范围和约束
- 参考框架
- 艺术/音频方向
- 风险评估
- 成功标准
- 下一步行动

## 比较：游戏简报 vs. GDD

| 方面                | 游戏简报                  | GDD                       |
| ------------------- | --------------------------- | ------------------------- |
| **目的**            | 验证概念                    | 为实现而设计               |
| **详细程度**        | 高层愿景                    | 详细规范                   |
| **时间投入**        | 1-2小时                     | 4-10小时                   |
| **受众**            | 自己、团队、利益相关者      | 开发团队                   |
| **范围**            | 概念验证                    | 实现路线图                 |
| **格式**            | 对话式、探索性              | 结构化、全面               |
| **输出**            | 3-5页                       | 10-30+页                   |

## 比较：游戏简报 vs. 产品简报

| 方面                | 游戏简报                   | 产品简报                     |
| ----------------- | ---------------------------- | --------------------------------- |
| **焦点**            | 玩家体验、乐趣、感受        | 用户问题、功能、价值              |
| **指标**            | 参与度、留存率、乐趣        | 收入、转化率、满意度              |
| **核心要素**        | 游戏支柱、机制              | 问题/解决方案、用户细分           |
| **参考**            | 其他游戏                    | 竞争对手、市场                    |
| **愿景**            | 情感体验                    | 业务成果                          |

## 示例用例

### 场景：独立Roguelike卡牌游戏

**起点：**
"我想制作一个带有转折的Roguelike卡牌游戏"

**游戏简报后：**

- **核心概念：** "一款Roguelike卡牌战斗游戏，您扮演情绪对抗内心恶魔"
- **目标受众：** 喜爱Slay the Spire的核心玩家，对心理健康主题感兴趣
- **差异化点：** 情绪叙事系统，其中卡组构成影响故事
- **MVP范围：** 3个角色、80张卡、30种敌人类型、3个Boss、6小时首次运行
- **平台：** 首选PC（Steam），后继移动设备
- **时间线：** 12个月，2人团队
- **关键风险：** 情绪主题可能疏远硬核Roguelike粉丝
- **缓解：** 早期原型，与目标受众测试，提供"仅机械"模式

**下一步行动：**

1. 构建卡牌战斗原型（2周）
2. 与玩家测试情感共鸣
3. 如果原型验证，继续进行GDD工作流程

## 成功技巧

### 对范围诚实

最常见的游戏开发失败是范围不匹配。使用此工作流程进行现实检查：

- 您的团队真的能构建这个吗？
- 时间线现实吗？
- 您有资产预算吗？

### 专注于玩家体验

不要考虑代码或实现。思考：

- 玩家会做什么？
- 他们会有什么感受？
- 他们为什么会关心？

### 早期验证

简报识别假设和风险。不要在没有以下情况下跳到GDD：

- 原型化风险机制
- 与目标受众测试
- 验证市场兴趣

### 明智使用参考

"像X但有Y"是起点，而不是差异化点。超越：

- 从参考游戏中具体是什么？
- 您明确不做的是什么？
- 什么是真正新的？

## 常见问题

**问：这是GDD之前必需的吗？**
答：不，但对新项目强烈推荐。如果您有清晰的愿景，可以直接开始GDD。

**问：我可以将其用于游戏马拉松吗？**
答：可以，但使用YOLO模式以加快速度。专注于愿景、机制和MVP范围。

**问：如果我的游戏概念改变了怎么办？**
答：重新审视和更新简报。在早期开发期间，它是一个活文档。

**问：内容量估算应该多详细？**
答：粗略的数量级就可以。"~50个敌人"而不是"47个敌人，每个有3个变体"。

**问：我应该单独完成还是与团队一起完成？**
答：让您的团队参与！协作简报能发现盲点并建立共同愿景。

## 相关工作流程

- **产品简报**（`1-analysis/product-brief`）：用于软件产品，而非游戏
- **GDD**（`2-plan/gdd`）：游戏简报后的下一步
- **叙事设计**（`2-plan/narrative`）：重故事游戏在GDD之后
- **解决方案设计**（`3-solutioning`）：规划后的技术架构
