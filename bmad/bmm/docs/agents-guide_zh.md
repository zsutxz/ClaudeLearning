## 概述

BMad Method 模块（BMM）提供了一支专业化的 AI 代理团队，引导您完成完整的软件开发生命周期。每个代理体现了具有独特专业知识、沟通风格和决策原则的特定角色。

**理念：** AI 代理充当专家协作者，而不是代码工具。他们带来数十年的模拟经验来指导战略决策、促进创造性思维并精确执行技术工作。

### 所有 BMM 代理

**核心开发（8个代理）：**

- PM（产品经理）
- 分析师（业务分析师）
- 架构师（系统架构师）
- SM（Scrum 主管）
- DEV（开发人员）
- TEA（测试架构师）
- UX 设计师
- 技术文档撰写者

**游戏开发（3个代理）：**

- 游戏设计师
- 游戏开发人员
- 游戏架构师

**元代理（1个核心代理）：**

- BMad 主控（协调者）

**总计：** 12个代理 + 跨模块派对模式支持

## 核心开发代理

### PM（产品经理）- John 📋

**角色：** 调查性产品策略师 + 精通市场的产品经理

**使用时机：**

- 为 2-4 级项目创建产品需求文档（PRD）
- 为小型项目（0-1 级）创建技术规范
- 将需求分解为史诗和故事
- 验证规划文档
- 实施过程中的路线修正

**主要阶段：** 阶段2（规划）

**工作流：**

- `workflow-status` - 检查下一步该做什么
- `create-prd` - 为 2-4 级项目创建 PRD
- `tech-spec` - 为 0-1 级项目快速制定规范
- `create-epics-and-stories` - 将 PRD 分解为可实施的部分
- `validate-prd` - 验证 PRD + 史诗的完整性
- `validate-tech-spec` - 验证技术规范
- `correct-course` - 处理项目中的变更
- `workflow-init` - 初始化工作流跟踪

**沟通风格：** 直接且分析性强。提出探究性问题以 uncover 根本原因。使用数据支持建议。对优先级和权衡精确表达。

**专业知识：**

- 市场研究和竞争分析
- 用户行为洞察
- 需求转化
- MVP 优先级排序
- 规模自适应规划（0-4 级）

### DEV（开发人员）- Amelia 💻

**角色：** 高级实施工程师

**使用时机：**

- 使用测试实施故事
- 对完成的故事进行代码审查
- 在完成定义满足后标记故事完成

**主要阶段：** 阶段4（实施）

**工作流：**

- `workflow-status` - 检查下一步该做什么
- `develop-story` - 实施故事，包括：
  - 任务迭代
  - 测试驱动开发
  - 多次运行能力（初始 + 修复）
  - 严格文件边界执行
- `code-review` - 高级开发人员级别审查，包括：
  - 故事上下文感知
  - 史诗技术上下文对齐
  - 仓库文档参考
  - MCP 服务器最佳实践
  - 网络搜索回退
- `story-done` - 标记故事完成并推进队列

**沟通风格：** 简洁和清单驱动。引用文件路径和验收标准ID。仅在输入缺失时提出问题。

**关键原则：**

- 故事上下文 XML 是唯一真实来源
- 在故事状态 == 已批准之前绝不开始
- 必须满足所有验收标准
- 完成前测试必须100%通过
- 不作弊或谎报测试结果
- 支持修复问题的多次运行

**专业知识：**

- 全栈实施
- 测试驱动开发（TDD）
- 代码质量和设计模式
- 现有代码库集成
- 性能优化

**使用时机：**

- 质量网关和可追溯性

**主要阶段：** 测试和QA（所有阶段）

**工作流：**

- `workflow-status` - 检查下一步该做什么

  - 智能框架选择（Playwright vs Cypress）
  - 夹具架构
  - 自动清理模式
  - 网络优先方法
- `atdd` - 首先生成E2E测试，实施前
- `automate` - 综合测试自动化
- `test-design` - 创建基于风险的测试场景
- `trace` - 需求到测试的可追溯性映射（阶段1 + 阶段2质量网关）
- `nfr-assess` - 验证非功能性需求
- `ci` - 搭建 CI/CD 质量管道

**沟通风格：** 数据驱动顾问。强烈的观点，弱持有。对权衡务实。

**原则：**

- 基于风险的测试（深度随影响扩展）

- 测试是功能工作，而不是开销
- 优先考虑单元/集成测试而不是E2E
- 不稳定性是关键技术债务
- ATDD测试优先，AI实施，套件验证

**特殊能力：**

- **知识库访问：** 从 `testarch/knowledge/` 目录咨询综合测试最佳实践
- **框架选择：** 智能框架选择（Playwright vs Cypress），配夹具架构
- **跨平台测试：** 支持跨web、移动和API层的测试

## 派对模式：多代理协作

将所有已安装的代理集中在一个对话中进行多视角讨论、回顾和协作决策。

**快速开始：**

```bash
/bmad:core:workflows:party-mode
# 或从任何代理：*party-mode
```

**发生什么：** BMad 主控编排每条消息2-3个相关代理。他们实时讨论、辩论和协作。

**最适合：** 战略决策、创造性头脑风暴、事后分析、冲刺回顾、复杂问题解决。

**当前BMM使用：** 驱动`epic-retrospective`工作流、冲刺规划讨论。

**未来：** 高级启发工作流将正式利用派对模式。

## 工作流访问

### 如何运行工作流

**从IDE（Claude Code、Cursor、Windsurf）：**

1. 使用代理引用加载代理（例如，在Claude Code中输入`@pm`）
2. 等待代理菜单出现在聊天中
3. 使用`*`前缀输入工作流触发器（例如，`*create-prd`）
4. 遵循工作流提示

**代理菜单结构：**
每个代理在加载时显示他们的可用工作流。查找：

- `*` 前缀表示工作流触发器
- 按类别或阶段分组
- 推荐入口点的START HERE指示器

### 通用工作流

一些工作流对多个代理可用：

| 工作流            | 代理                            | 目的                                     |
| ----------------- | ------------------------------- | ---------------------------------------- |
| `workflow-status` | 所有代理                        | 检查当前状态并获取建议                   |
| `workflow-init`   | PM、分析师、游戏设计师          | 初始化工作流跟踪                         |
| `correct-course`  | PM、架构师、SM、游戏架构师      | 实施过程中的变更管理                     |
| `document-project` | 分析师、技术文档撰写者          | 棕地文档                                 |

### 验证操作

许多工作流有可选的验证工作流执行独立审查：

| 验证                    | 代理         | 验证                              |
| ----------------------- | ------------ | --------------------------------- |
| `validate-prd`          | PM           | PRD + 史诗 + 故事完整性           |
| `validate-tech-spec`    | PM           | 技术规范质量                       |
| `validate-architecture` | 架构师       | 架构文档                           |
| `validate-design`       | UX 设计师    | UX规范和工件                      |
| `validate-epic-tech-context` | SM      | 史诗技术上下文                    |
| `validate-create-story` | SM           | 故事草稿                           |
| `validate-story-context`| SM           | 故事上下文 XML                     |

**何时使用验证：**

- 阶段转换前
- 关键文档
- 学习BMM时
- 高风险项目

## 代理定制

您可以在不修改核心代理文件的情况下定制任何代理的个性。

### 位置

**定制目录：** `{project-root}/bmad/_cfg/agents/`

**命名约定：** `{module}-{agent-name}.customize.yaml`

**示例：**

```
bmad/_cfg/agents/
├── bmm-pm.customize.yaml
├── bmm-dev.customize.yaml
├── cis-storyteller.customize.yaml
└── bmb-bmad-builder.customize.yaml
```

### 覆盖结构

**文件格式：**

```yaml
agent:
  persona:
    displayName: '自定义名称' # 可选：覆盖显示名称
    communicationStyle: '自定义风格描述' # 可选：覆盖风格
    principles: # 可选：添加或替换原则
      - '此项目的自定义原则'
      - '另一个项目特定指导方针'
```

### 覆盖行为

**优先级：** 定制 > 清单

**合并规则：**

- 如果在定制中指定字段，它替换清单值
- 如果未指定字段，使用清单值
- 额外字段添加到代理个性
- 当代理加载时更改立即生效

### 用例

**调整正式性：**

```yaml
agent:
  persona:
    communicationStyle: '正式和企业聚焦。使用商业术语。结构化响应配执行摘要。'
```

**添加领域专业知识：**

```yaml
agent:
  persona:
    identity: |
      在医疗保健SaaS领域有15年经验的专业产品经理。
      深刻理解HIPAA合规性、EHR集成和临床工作流。
      专注于平衡监管要求与用户体验。
```

**修改原则：**

```yaml
agent:
  persona:
    principles:
      - 'HIPAA合规性不可协商'
      - '患者安全优先于功能速度'
      - '每个功能必须有临床验证'
```

**更改个性：**

```yaml
agent:
  persona:
    displayName: 'Alex' # 从默认"Amelia"更改
    communicationStyle: '随意和友好。使用表情符号。用简单术语解释技术概念。'
```

### 派对模式集成

定制自动在派对模式中应用：

1. 派对模式读取清单
2. 检查定制文件
3. 将定制与清单合并
4. 代理以定制个性响应

**示例：**

```
您用医疗保健专业知识定制PM。
在派对模式中，PM现在为讨论带来医疗保健知识。
其他代理与PM的专业视角协作。
```

### 应用定制

**重要：** 定制在您重建代理之前不会生效。

**完整过程：**

**步骤1：创建/修改定制文件**

```bash
# 在以下位置创建定制文件：
# {project-root}/bmad/_cfg/agents/{module}-{agent-name}.customize.yaml

# 示例：bmad/_cfg/agents/bmm-pm.customize.yaml
```

**步骤2：重新生成代理清单**

修改定制文件后，您必须重新生成代理清单并重建代理：

```bash
# 运行安装程序应用定制
npx bmad-method install

# 安装程序将：
# 1. 读取所有定制文件
# 2. 用合并的数据重新生成agent-manifest.csv
# 3. 用应用的定制重建代理.md文件
```

**步骤3：验证更改**

加载定制代理并验证更改反映在其行为和响应中。

**为什么这是必需的：**

- 定制文件只是配置 - 它们不直接更改代理
- 必须重新生成代理清单以合并定制
- 必须用合并的数据重建代理.md文件
- 派对模式和所有工作流从重建的文件加载代理

### 最佳实践

1. **保持项目特定性：** 为您的领域定制，而不是一般更改
2. **不破坏角色：** 保持定制与代理核心角色一致
3. **在派对模式中测试：** 查看定制如何与其他代理交互
4. **记录原因：** 添加注释解释定制目的
5. **与团队分享：** 定制在更新中幸存，可以版本控制
6. **更改后重建：** 修改定制文件后始终运行安装程序

## 最佳实践

### 代理选择

**1. 从workflow-status开始**

- 当不确定位置时，加载任何代理并运行`*workflow-status`
- 代理将分析当前项目状态并推荐下一步
- 在所有阶段和所有代理中工作

**2. 将阶段与代理匹配**

- **阶段1（分析）：** 分析师、游戏设计师
- **阶段2（规划）：** PM、UX设计师、游戏设计师
- **阶段3（解决方案设计）：** 架构师、游戏架构师
- **阶段4（实施）：** SM、DEV、游戏开发人员
- **测试：** TEA（所有阶段）
- **文档：** 技术文档撰写者（所有阶段）

**3. 使用专家**

- **测试：** TEA用于综合质量策略
- **文档：** 技术文档撰写者用于技术写作
- **游戏：** 游戏设计师/开发人员/架构师用于游戏特定需求
- **UX：** UX设计师用于以用户为中心的设计

**4. 尝试派对模式用于：**

- 有权衡的战略决策
- 创造性头脑风暴会议
- 跨职能对齐
- 复杂问题解决

### 与代理协作

**1. 信任他们的专业知识**

- 代理体现数十年的模拟经验
- 他们的问题 uncover 关键问题
- 他们的建议是数据知情的
- 他们的警告防止代价高昂的错误

**2. 回答他们的问题**

- 代理出于重要原因提问
- 不完整的答案导致假设
- 详细的响应产生更好的结果
- "我不知道"是有效答案

**3. 遵循工作流**

- 结构化流程防止遗漏步骤
- 工作流编码最佳实践
- 顺序工作流相互构建
- 验证工作流及早发现错误

**4. 需要时定制**

- 为您的项目调整代理个性
- 添加领域特定专业知识
- 为团队偏好修改沟通风格
- 保持定制项目特定

### 常见工作流模式

**开始新项目（绿地）：**

```
1. PM或分析师：*workflow-init
2. 分析师：*brainstorm-project或*product-brief（可选）
3. PM：*create-prd（2-4级）或*tech-spec（0-1级）
4. 架构师：*create-architecture（仅3-4级）
5. SM：*sprint-planning
```

**从现有代码开始（棕地）：**

```
1. 分析师或技术文档撰写者：*document-project
2. PM或分析师：*workflow-init
3. PM：*create-prd或*tech-spec
4. 架构师：*create-architecture（如需要）
5. SM：*sprint-planning
```

**故事开发周期：**

```
1. SM：*epic-tech-context（可选，每个史诗一次）
2. SM：*create-story
3. SM：*story-context
4. DEV：*develop-story
5. DEV：*code-review
6. DEV：*story-done
7. 为下一个故事重复步骤2-6
```

**测试策略：**

```
1. TEA：*framework（每个项目一次，早期）
2. TEA：*atdd（实施功能前）
3. DEV：*develop-story（包括测试）
4. TEA：*automate（综合测试套件）
5. TEA：*trace（质量网关）
6. TEA：*ci（管道设置）
```

**游戏开发：**

```
1. 游戏设计师：*brainstorm-game
2. 游戏设计师：*create-gdd
3. 游戏架构师：*create-architecture
4. SM：*sprint-planning
5. 游戏开发人员：*create-story
6. 游戏开发人员：*dev-story
7. 游戏开发人员：*code-review
```

### 导航技巧

**迷路了？运行workflow-status**

```
加载任何代理 → *workflow-status
代理分析项目状态 → 推荐下一个工作流
```

**阶段转换：**

```
每个阶段都有验证网关：
- 阶段2→3：validate-prd、validate-tech-spec
- 阶段3→4：solutioning-gate-check
在前进前运行验证
```

**路线修正：**

```
如果项目中途优先级变化：
加载PM、架构师或SM → *correct-course
```

**测试集成：**

```
TEA可以在任何阶段调用：
- 阶段1：测试策略规划
- 阶段2：PRD中的测试场景
- 阶段3：架构可测试性审查
- 阶段4：测试自动化和CI
```
